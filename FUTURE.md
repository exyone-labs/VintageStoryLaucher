## 一、项目现状分析

### 1.1 技术架构概览

VSL 是一个基于 WPF (.NET 9) 构建的 Vintage Story 游戏服务器启动器，采用经典四层架构设计：

| 层次 | 模块 | 代码行数(估) | 核心职责 |
|------|------|--------------|----------|
| 表现层 | VSL.UI | ~2500 | WPF 界面与 MVVM 实现 |
| 应用层 | VSL.Application | ~300 | 接口定义与操作结果模式 |
| 领域层 | VSL.Domain | ~800 | 实体与值对象定义 |
| 基础设施层 | VSL.Infrastructure | ~2000 | 服务实现与数据持久化 |

### 1.2 现有优势

- **技术栈现代**: .NET 9 + WPF-UI (Fluent Design)，语法现代 (records, primary constructors)
- **依赖注入完善**: 全面使用 Microsoft.Extensions.DependencyInjection
- **异步编程规范**: 贯穿全项目的 async/await，无阻塞 UI
- **测试覆盖**: 包含单元测试项目，使用 xUnit 框架
- **操作结果模式**: 统一的成功/失败处理范式
- **CI/CD 成熟**: GitHub Actions 自动构建与发布

### 1.3 存在的主要问题

#### 问题 1: MainViewModel 过于庞大 (高优先级)

**现状**: `MainViewModel` 构造函数超过 300 行，包含：
- 13 个服务依赖注入
- 20+ 个命令定义
- 40+ 个属性定义
- 所有业务逻辑集中处理

**影响**:
- 代码可维护性差
- 难以进行单元测试
- 新功能开发效率低
- 并行开发冲突频繁

#### 问题 2: 缺乏清晰的事件通信机制 (中优先级)

**现状**: ViewModel 与 View 之间、ViewModel 与 ViewModel 之间通过直接引用或属性绑定通信

**影响**:
- 模块耦合度高
- 难以独立测试各功能模块
- 功能扩展需要修改现有代码

#### 问题 3: 缺少文件系统抽象 (中优先级)

**现状**: 服务直接使用 `System.IO` 操作文件系统

**影响**:
- 单元测试需要实际文件系统
- 难以模拟异常场景
- 跨平台支持成本高

#### 问题 4: 配置与硬编码混合 (中优先级)

**现状**: 部分配置逻辑分散在代码中，如路径分隔符、默认值等

**影响**:
- 配置变更需要重新编译
- 多环境部署不灵活

#### 问题 5: 错误处理不一致 (低优先级)

**现状**: 各服务对异常的处理粒度不统一，部分直接抛出、部分返回失败结果

**影响**:
- 上层调用难以统一处理
- 错误信息不友好

## 二、优化目标

### 2.1 总体目标

在保持现有功能稳定的前提下，通过架构优化提升代码的可维护性、可测试性和可扩展性，为后续功能迭代奠定坚实基础。

### 2.2 具体目标

| 目标类别 | 目标项 | 当前状态 | 目标状态 | 优先级 |
|----------|--------|----------|----------|--------|
| 可维护性 | MainViewModel 复杂度 | 单一文件 ~2000 行 | 拆分为 6-8 个子 ViewModel | P0 |
| 可测试性 | 服务可测试性 | 部分服务难以测试 | 90% 服务可单元测试 | P0 |
| 可扩展性 | 插件机制 | 无 | 支持轻量级扩展 | P1 |
| 性能 | 启动时间 | 未测量 | 减少 20% | P2 |
| 质量 | 测试覆盖率 | ~30% | 提升至 60%+ | P1 |

### 2.3 量化指标

| 指标 | 基线 | 目标 | 测量方法 |
|------|------|------|----------|
| MainViewModel 代码行数 | ~2000 行 | < 300 行 | 代码统计工具 |
| 测试覆盖率 | ~30% | ≥ 60% | dotCover / Coverage.py |
| 单元测试数量 | ~50 | ≥ 150 | 测试框架报告 |
| 关键服务可测试率 | ~60% | ≥ 90% | 手动评估 |
| 平均构建时间 | ~60 秒 | < 45 秒 | CI/CD 统计 |

## 六、风险评估及应对策略

### 6.1 风险识别

| 风险 ID | 风险描述 | 概率 | 影响 | 等级 |
|---------|----------|------|------|------|
| R1 | 重构导致功能退化 | 高 | 高 | **严重** |
| R2 | 进度延期 | 中 | 中 | 中等 |
| R3 | 团队人员变动 | 低 | 高 | 中等 |
| R4 | 依赖库版本冲突 | 中 | 中 | 低 |
| R5 | 测试覆盖目标难以达成 | 中 | 中 | 低 |
| R6 | 性能优化未达预期 | 中 | 低 | 低 |

### 6.2 应对策略

#### R1: 重构导致功能退化

**预防措施**:
- 严格的代码审查 (Code Review)
- 每次重构后运行完整测试套件
- 小步迭代，每次修改后验证功能

**应急预案**:
- 保留功能分支，必要时回滚
- 每日集成测试，及早发现问题

#### R2: 进度延期

**预防措施**:
- 详细的阶段计划与里程碑
- 每周进度评审与调整
- 预留 15% 时间缓冲

**应急预案**:
- 拆分非关键任务到后续阶段
- 增加资源投入 (如可行)

#### R3: 团队人员变动

**预防措施**:
- 详细的文档记录
- 代码注释与规范
- 知识分享会议

**应急预案**:
- 交接文档与培训
- 关键模块双人负责制

#### R4: 依赖库版本冲突

**预防措施**:
- 使用 Directory.Packages.props 统一版本
- 定期更新依赖并测试

**应急预案**:
- 锁定关键依赖版本
- 预留时间进行依赖升级测试

#### R5: 测试覆盖目标难以达成

**预防措施**:
- 优先覆盖核心业务逻辑
- 自动化测试生成工具

**应急预案**:
- 调整目标阈值
- 聚焦关键路径测试

#### R6: 性能优化未达预期

**预防措施**:
- 科学的性能基线测量
- Profiling 工具辅助定位

**应急预案**:
- 接受当前性能，接受合理优化
- 记录优化经验供后续参考
